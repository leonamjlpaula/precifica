## Codebase Patterns
- Dashboard layout: server component (session check + redirect) wraps a client component (DashboardLayout) that handles nav, sidebar, and mobile bottom nav
- Active nav state: use `'/' + href.split('/')[1]` vs `'/' + pathname.split('/')[1]` to match base segment (handles nested routes like /procedimentos/diagnostico)
- Mobile nav "Mais" pattern: fixed bottom overlay grid with remaining nav items; close on backdrop click or item click
- getServerSession(authOptions) in server components for session access; signOut({ callbackUrl: '/login' }) in client components for logout
- useActionState from React 19 (not useFormState) for server action form state in client components
- Auth pages go in src/app/(auth)/ route group
- Server actions go in src/application/usecases/ as 'use server' files
- On successful form submission, return { success: true } and use useEffect + router.push() for client-side redirect (server-side redirect() doesn't work well with useActionState)
- Export both the action function AND the state type from server action files so client components can type the initialState


- next-auth@4 stable: authOptions in src/lib/auth.ts, API route at app/api/auth/[...nextauth]/route.ts, session JWT strategy
- next-auth type augmentation in src/types/next-auth.d.ts (extends Session.user with id: string, JWT with userId)
- Middleware at src/middleware.ts uses withAuth from next-auth/middleware — protect /dashboard/:path*
- session.user.id is populated via jwt callback (token.userId) → session callback (session.user.id = token.userId)
- Next.js 15.5.12 with App Router, TypeScript strict mode
- shadcn/ui configured manually via components.json (create-next-app won't work in existing directory with files)
- UI components live in src/presentation/components/ui/ (aliased as @/presentation/components/ui)
- All utilities in src/lib/utils.ts (cn helper)
- Hooks in src/presentation/hooks/
- Prisma schema at prisma/schema.prisma with postgresql provider
- Domain structure: src/domain/entities, src/domain/value-objects, src/application/usecases, src/application/interfaces, src/infrastructure/repositories, src/infrastructure/services
- Run typecheck with: npx tsc --noEmit
- next-auth@4 stable used (not v5 beta)
- bcrypt for password hashing (@types/bcrypt in devDependencies)
- No local PostgreSQL available — use `prisma migrate diff --from-empty --to-schema-datamodel prisma/schema.prisma --script` to generate migration SQL; run `npx prisma migrate dev` when DB is available
- Migration files committed to prisma/migrations/ so they're ready for deployment
- Prisma singleton at src/lib/db.ts — import `prisma` from '@/lib/db' in server code
- VRPO seed data (materials, procedures, createDefaultDataForUser) in src/lib/vrpo-seed-data.ts
- Global seed (Especialidades + VRPOReferencias) in prisma/seed.ts — run `npx prisma db seed` when DB ready
- seed.ts uses its own PrismaClient directly (not the singleton from db.ts) to avoid module resolution issues in ts-node
- prisma/seed.ts does NOT import from src/lib/* to avoid @/ alias resolution issues in ts-node CommonJS mode
